## About git push deployment

Git push deployment means that the same `git push` that sends your commits to GitHub also updates your live website. Compared to uploading files via FTP, every deployment is repeatable and versioned: you always know which code is live, every team member deploys the same way, and rolling back means pushing a previous state.

There is no special Git remote to configure here. You push to GitHub like you always do — the [fortrabbit GitHub App](/1.platform/05.deployment/02.github-app.md) watches the connected branch and takes it from there:

1. You commit and push to GitHub.
2. fortrabbit picks up the push and pulls the latest code.
3. Build commands run, for example `composer install`.
4. The result is synced into the web space of your [environment](/1.platform/10.objects/02.environment.md).

The [deployment intro](/1.platform/05.deployment/01.intro.md) covers the full pipeline in detail.

## Get ready

- Have [Git installed](/3.dev/15.learn/git-intro.md) on your local machine.
- Have a [GitHub](https://github.com) account.
- Have a PHP project — plain PHP is fine, frameworks and CMSs work too.
- Have a fortrabbit account, the :ContentLink{href="/signup" text="free trial" prefix="dash"} is enough.

## Step 1: Put your project in a Git repo

Skip ahead if your project is already version controlled. Otherwise, open a terminal in your project folder:

```shell
$ git init
$ git add .
$ git commit -m "Initial commit"
```

Before the first commit, add a `.gitignore` file so generated files and secrets stay out of the repo:

```raw
/vendor/
.env
```

The `vendor` folder does not belong in Git — dependencies get installed during deployment. See the [Composer article](/3.dev/15.learn/02.composer.md) for the reasoning.

## Step 2: Push it to GitHub

Create a new repository on GitHub, then connect it to your local repo and push:

```shell
$ git remote add origin git@github.com:{{your-name}}/{{your-repo}}.git
$ git push -u origin main
```

You can also start the other way around: when creating an app on fortrabbit, the GitHub App can create a fresh empty repo for you.

## Step 3: Connect the repo to fortrabbit

Sign up — or log in — with GitHub, which installs the [fortrabbit GitHub App](/1.platform/05.deployment/02.github-app.md) on your GitHub account. Then create an [app](/1.platform/10.objects/01.app.md) and pick your repository and branch:

- The Git repo maps to the app.
- The Git branch maps to the environment — `main` for production, other branches for [staging](/3.dev/15.learn/03.staging-environments.md).

During setup, fortrabbit inspects the repo and preconfigures the deployment: when it finds a `composer.json`, the [build commands](/1.platform/05.deployment/03.build-commands.md) will include `composer install`.

## Step 4: Deploy with a git push

That was the one-time setup. From now on, deployment is part of your normal Git routine:

```shell
$ git commit -am "Update homepage"
$ git push
```

Push to deploy is the default [deployment trigger](/1.platform/05.deployment/05.trigger.md): every push to the connected branch starts a deployment automatically. Follow the progress in the [deployment logs](/1.platform/05.deployment/17.logs.md) in the dashboard, then open your app URL in the browser to see the change live.

## What gets deployed — and what not

Git only moves code: templates, classes, themes, CSS, JS. Uploads, the database, and other runtime data are not part of the deployment and stay untouched on the environment. Git is also a one-way street here — you can not pull changes from the web space back into your repo. The [deployment intro](/1.platform/05.deployment/01.intro.md) explains this separation of code and content.
