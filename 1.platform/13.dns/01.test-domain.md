## Structure

Each environment gets a unique test domain following this pattern:

```text
{{ environment }}.{{ region }}.frbit.app
```

- **environment** — The ID of the environment, a short, randomly generated unique identifier. It is set when the environment is created and cannot be changed afterward.
- **region** — the data center region where the environment is hosted, for example `us1` or `eu2`.
- **frbit.app** — the fortrabbit root domain used for all test domains.

- Example: `https://my-app-abc123.eu2.frbit.app`
- Live code example: `{{app-env-id}}.{{region}}.frbit.app`

- TLS is included. The test domain is always served over HTTPS.

## Use cases

The test domain is available immediately after an environment is created — no DNS configuration required. Common uses:

- **Quick testing** — open the app in a browser right away after a deploy without any setup.
- **Sharing work in progress** — send the test domain to a colleague or client for review before going live.
- **Staging environments** — use the test domain for a staging or preview environment that does not need a public-facing custom domain.
- **CI/CD pipelines** — reference the test domain in automated tests or health checks.

## Limits

- **The name cannot be changed.** The app-env-id part of the test domain is fixed at environment creation time. If the identifier does not suit you, the only way to get a new one is to create a new environment.
- The test domain is not a replacement for a production domain. For a customer-facing address, add an [external domain](/1.platform/13.dns/02.external-domains.md).
