## About logs

Every environment collects logs from the PHP runtime, the web server and background jobs. They are the first place to look when a request fails, a page turns white or traffic behaves unexpectedly.

## Accessing logs

Logs are available in the dashboard under the environment's **Logs** tab. There is no need to log in via SSH to read them.

:BlockLink{title="See logs for {{app-name}} / {{app-env-name}}" path="/environments/{{app-env-id}}/logs"}

## Log sources

The dashboard groups logs into separate sources. Switch between them with the source selector. Only output written to `stderr` and `stdout` is collected — see [Log files in your CMS or framework](#log-files-in-your-cms-or-framework) if a source stays empty.

### PHP logs

Application-level output from the PHP runtime. Uncaught exceptions, fatal errors, warnings and notices end up here, along with anything the code sends through `error_log()`. Each line carries a severity — from `NOTICE` and `WARNING` up to `ERROR` — so the level filter narrows a noisy log down to what actually broke. This is the first place to look for a white screen of death or an HTTP 500.

### Access logs

One line per HTTP request, in the Apache combined log format: client IP, timestamp, request method and path, response status code, response size, referrer and user agent. Use it to watch traffic in real time, spot which URLs return `404`, or trace a spike of `500` responses back to a single client. Method, status and IP address are all filterable.

### Apache error logs

The web server's own error log, separate from application errors. It records problems Apache runs into while handling a request: rewrite rule failures, file permission issues, denied directory access and requests the server could not fulfil. Lines carry a syslog severity (`warn`, `error`, `crit` …) and, where Apache attributes one, an error code such as `AH01630`. Look here when a URL misbehaves but nothing shows up in the PHP log.

### Jobs logs

Output from background jobs and scheduled tasks running in the [Jobs component](/1.platform/03.concepts/03.components.md): worker processes and cron-style commands. This source only appears when the Jobs component is enabled. Use it to confirm a scheduled task ran and to read what it printed.

## Browsing and filtering

The log view streams lines newest first and can be narrowed down without leaving the page:

- **Filter by level** — limit PHP, Apache error and Jobs logs to a severity such as `ERROR` or `WARNING`.
- **Filter by method and status** — for access logs, narrow down to a request method (`GET`, `POST`, …) or an HTTP status code (`404`, `500`, …).
- **Filter by IP** — click an IP address in an access or Apache error line to filter for it, or look it up on ipinfo.io.
- **Search** — free-text search across the visible lines.
- **Pick a time range** — jump to a preset like today or yesterday, or define a custom day and hour. Times are UTC.
- **Follow live** — keep the latest lines streaming in as new requests come in.
- **Load more** — page further back through the history.

Clicking a level, method or status code in a line filters for that value, so drilling into one kind of event takes a single click.

## Downloading logs

To keep logs or analyze them elsewhere, switch to **Download**, pick a source and a time range, and request a download link. The export is queued and a link is mailed once it is ready. Larger ranges take a few minutes to prepare.

## Tailing logs over SSH

The browser log view above is the recommended way to read logs. Live streaming is also available in the terminal:

1. Log in to the environment by [SSH](/1.platform/04.code-access/03.ssh.md)
2. Run `frbit-tail-logs`

This tails `stderr` and `stdout` as a live stream.

## Log files in your CMS or framework

A CMS or framework might pipe logs to a different location, other than `stderr` and `stdout`. In that case the output does not show up in the dashboard. The logs can still be read by [SSH](/1.platform/04.code-access/03.ssh.md) or [SFTP](/1.platform/04.code-access/04.sftp.md) — see the CMS / framework docs for where they are stored.

Most CMS and frameworks also offer options to send logs to the standard locations. With the [software template](/1.platform/03.concepts/05.software-templates.md) the setting is pre-populated by ENV var where possible.

- [Craft CMS logging](/2.guides/3.craft-cms/06.tune.md#logging)
- [Laravel logging](/2.guides/2.laravel/05.tune.md#logging)

### Verbose logging

A CMS / framework might offer verbose logging, often tied to its environment settings. This prints more debugging information with each error, and sometimes sends errors to the browser. Use it with care, as it can have a considerable performance impact. The general advice is to use verbose logging on development and staging environments, but not in production. If an issue in production requires it, turn it off again once finished.
